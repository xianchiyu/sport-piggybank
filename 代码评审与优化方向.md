# 运动存钱罐 — 代码质量评审与优化方向

> 评审人：Senior Developer（高级开发工程师）
> 评审范围：`sport-piggybank` 全量源码（Kotlin 桥接层 + 前端 SPA + 构建/CI）
> 目标：定位优化方向，作为团队技术能力提升的抓手

---

## 一、总体评价（先说好的）

这个项目作为 MVP 完成度很高，方向也对：

- ✅ **分层意识已有雏形**：`PiggyCore.kt` 把币值计算（`CoinUtils`）独立出来了，纯函数、好测。
- ✅ **WebView 壳极简**（~3MB），桥接协议 `{ok, data/error}` 统一返回，前端解析简单。
- ✅ **前端无框架、零依赖**，纯原生 DOM，体积小、可离线。
- ✅ **CI 自动出包**、`minSdk 24` / `targetSdk 34` 设定合理。
- ✅ 业务规则（连续加成、自动合成、三角数罚金）实现清晰，算法可读。

但要从「能跑」走到「团队能长期维护、可扩展、可信赖」，下面这些问题是当前最大的杠杆点。

## 二、优化路线图（按优先级）

### 🔴 P0 — 必须修：正确性缺陷

| # | 问题 | 位置 | 影响 |
|---|---|---|---|
| 1 | **自动违规「检测」与「惩罚」两张代码、互不一致** | `MainActivity.getAutoViolations()`(L281-323) 只返回违规列表、不执行惩罚；`PiggyCore.AutoPenalty.check()`(L183-237) 才是惩罚逻辑，但**从未被调用**。结果：自动惩罚实际没生效，只有手动上报才罚。且两段逻辑重复、必然漂移。 | 核心规则失效 + 死代码 |
| 2 | **趋势图数据契约不一致** | `renderTrendChart()`(app.js L296-332) 读 `t.coinChange`，但真实交易（`addTransaction` L344-358）只存 `date/type/subtype/amount/note/time`，**没有 coinChange**。图表静默 fallback 到 amount 估算，且从不按铜→银→金合成，曲线数值错误。 | 统计页展示错误 |
| 3 | **`spend()` 金币路径计算脆弱** | `CoinUtils.spend()`(L122-153) 在拆金币分支用 `silver = goldNeed*10 - silverShort` 后重算找零，逻辑绕、易 off-by-one，且**无任何测试覆盖**。 | 消费拆币可能算错 |
| 4 | **Getter 带副作用（反模式）** | `getSocialExemptStatus()`(L210-221)、`getAutoViolations()`(L281) 在「读取」时偷偷重置月度计数 / 写入检测日期。并发或重复调用会改状态。 | 隐蔽 bug、不可测 |

**P0 建议**：先统一自动违规逻辑（合并为 `AutoPenalty.evaluate()` 一处，既检测又执行），并为 `CoinUtils` 全量补单测（见 P1）。

### 🟠 P1 — 架构与可测试性（团队成长核心）

这是提升团队水平**最高 ROI** 的一块。

**1. 业务核心逻辑不应写在 `JsBridge` 里**
`MainActivity.JsBridge` 的 `checkin / consume / reportViolation / quarterWithdraw`(L72-273) 把领域逻辑、状态读写、JSON 序列化全耦合在一个 WebView 内部类里。
→ **目标**：抽出独立 `PiggyEngine`（纯 Kotlin，不依赖 Android Context），桥接层只做「参数 → 调用 → JSON 返回」。
```
当前：  JsBridge.checkin() { 读 PiggyData → 算币 → 写 PiggyData → 拼 JSON }
目标：  JsBridge.checkin() { engine.checkin(...) → toJson(result) }
        PiggyEngine.checkin() { 纯逻辑，可单测 }
```

**2. 持久化从 SharedPreferences 升级到 Room**
`PiggyData` 用 SP 逐字段 `apply()`，且 `transactions` 存成**一个 JSON 字符串**，每次插入都整体重建数组并截断到 200 条（L344-358，O(n) 每次）。
→ 项目 spec 本就规划了 SQLite。建议用 **Room**：`Transactions` 成独立表（结构化 `type/subtype/coinChange`），查询/分页/合并都变简单，也为多用户、报表、导出铺路。

**3. 金额用 `Float` 有精度风险**
`quarterIncome / quarterExpense` 是 `Float`，累加会产生浮点误差（金钱相关）。
→ 存「分」用 `Int`，或 `BigDecimal`。币数本身是 `Int`（好），只有元是 Float。

**4. 前端把结构化数据丢掉了**
真实交易带 `type/subtype`，但前端 `updateCheckStatus()`(L128-146)、`renderCheckinGrid()`(L281-287) 都靠**对中文 note 做 `indexOf('运动')` 子串匹配**来还原状态。文案一改就崩。
→ 桥接返回里直接带结构化标记（如 `t.subtype === 'exercise'`），前端按字段判断，删除所有子串猜测。

**5. `PiggyCore` 与 `MainActivity` 职责重叠**
`AutoPenalty`、`WeatherHelper` 在 PiggyCore，`checkin` 主逻辑在 MainActivity。建议统一收口到一个 `core` 包，目录清晰：
```
core/
  model/   (PiggyState, Transaction, 枚举)
  engine/  (PiggyEngine, CoinUtils, AutoPenalty)
  infra/   (Storage 接口 → Room 实现, WeatherClient)
bridge/    (JsBridge 只做序列化)
```

### 🟡 P2 — 工程化与质量门禁

| # | 现状 | 建议 |
|---|---|---|
| 1 | **零自动化测试** | 加 `test` 源集 + JUnit5/AssertJ，`CoinUtils` 全函数覆盖（autoMerge/spend/cashPenalty/multiplier/exerciseCoins）。这是团队最该养成的习惯。 |
| 2 | CI 只 `assembleDebug`，无 lint/测试 | CI 加 `./gradlew test` + **detekt** + **ktlint**；把"每次 push 都发 GitHub Release"改为仅上传 Artifact，Release 走手动/Tag。 |
| 3 | `tech-spec.md` 描述的是**微信小程序+云开发**，与现状（安卓 WebView）严重不符 | 更新或归档，避免新人被误导。文档漂移是团队认知负债。 |
| 4 | `Calendar` / `SimpleDateFormat` 旧 API，且 `daysBetween` 用整数除毫秒（跨夏令时会差 1 天） | 启用 **core library desugaring** 后用 `java.time`（`LocalDate`、`ChronoUnit.DAYS`）。 |
| 5 | `WeatherHelper.isRainy()` 在 `checkin` 路径里**同步发 HTTP**（5s 超时） | 首次点击打卡可能卡 5 秒。改为：启动预取并缓存、或桥接异步返回；至少用已缓存值兜底。 |

### 🟢 P3 — 体验与「高级感」（契合团队对品质的要求）

> 团队规范里明确要求前端支持**深浅色主题切换**，当前只有浅色奶油主题。

| # | 方向 | 做法 |
|---|---|---|
| 1 | **深色模式** | CSS 变量化配色（`--bg/--card/--text`），桥接暴露 `getTheme/setTheme` 存 SP，首次按系统 `prefers-color-scheme` 取。这是签名级升级。 |
| 2 | `app.js` 是 550 行全局函数 + `window.Android` 到处分支 | 拆 ES Module（state / api-bridge / pages），用 esbuild 打包；浏览器里注入 mock bridge，使同一份代码双端运行，去掉所有 `if (window.Android)`。 |
| 3 | 可访问性（WCAG 2.1 AA） | 按钮加 `aria-label`、可见 focus 态、图表加 `<title>`/`role=img`。 |
| 4 | WebView 安全加固 | 当前 `allowFileAccess=true` + `addJavascriptInterface` + `file://`。本地无远端内容风险低，但正确做法是 `WebViewAssetLoader` 以 `https` scheme 提供 assets 并关掉 `allowFileAccess`；`onDestroy` 调 `webView.destroy()` 防泄漏。 |
| 5 | 动画/微交互 | 已有投币 drop 动画；可加连续天数进度环、季度进度条，强化"养成"反馈。 |

## 四、建议的第一个冲刺（落地、可演示）

按"先止血、再建骨架、后打磨"的顺序，推荐这样排：

1. **P0-1 合并自动违规逻辑** → 一处 `AutoPenalty.evaluate()`，删掉重复代码。半天。
2. **P1-5 + P0-3 给 `CoinUtils` 写单测**（autoMerge/spend/cashPenalty/multiplier/exerciseCoins），覆盖边界。半天。**这一步价值最大：团队第一次有测试。**
3. **P1-1 抽 `PiggyEngine`**，把 `JsBridge` 里的逻辑搬出来，桥接层瘦身。1 天。
4. **P1-4 前端改用 `type/subtype` 结构化判断**，删子串匹配。半天。
5. **P2-2 CI 加 detekt + test**，把"发 Release"改成 Artifact。2 小时。
6. （可选）**P3-1 深色模式** —— 最能让团队/用户感受到"品质"的改动。

## 五、重点提醒（容易踩的坑）

- ⚠️ **`quarterWithdraw()`(L258) 直接把金币清零**，但没有对账/历史留存（spec 有 `quarterly` 表但未实现）。提现前应先写一条 `withdraw` 交易并保留季度汇总，否则数据不可追溯。
- ⚠️ **`transactions` 存 JSON 字符串 + 200 条上限**：长期用会丢历史，报表/趋势都不准。Room 化是根本解。
- ⚠️ **`allowBackup="true"`**（Manifest L10）：本地理财数据会被 Google 备份，隐私角度建议 `false` 或自定义 backup 规则。
